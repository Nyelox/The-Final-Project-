Main Execution Function (with Hybrid Encryption Handshake)

def main(): # ===== Hybrid encryption key exchange with the server ===== # 1. Fetch the server's RSA public key r = requests.get(f"{SERVER_URL}/public_key", timeout=5) pem = r.json()["public_key"] public_key = crypto_utils.import_public_key(pem) # 2. Generate a random AES session key aes_key = crypto_utils.generate_aes_key() # 3. Encrypt the AES key with the server's RSA public key encrypted_key = crypto_utils.rsa_encrypt(public_key, aes_key) # 4. Send the encrypted key, receive a session_token r2 = requests.post(f"{SERVER_URL}/session_key", json={"encrypted_key": encrypted_key}, timeout=5) session_token = r2.json()["session_token"] # Store everything for all future requests crypto_utils.AES_KEY = aes_key crypto_utils.SESSION_TOKEN = session_token crypto_utils.SERVER_PUBLIC_KEY = public_key print("Encrypted session established with server") app = QApplication(sys.argv) login_window = Login() login_window.show() sys.exit(app.exec_()) if __name__ == "__main__": main()
Massive Detailed Explanation:
This is the absolute starting point of the application. Note that the encryption handshake runs BEFORE any UI is drawn. If the handshake fails, the app will never open the login window — by design, no plaintext data should ever leave the laptop.

The 4-Step Handshake:
1. Fetch the server's RSA public key via GET /public_key. This is the only request that is sent unencrypted — public keys are by definition public.
2. Generate a random 32-byte AES key locally with os.urandom. Only we know it.
3. Encrypt the AES key with RSA-OAEP-SHA256. The result can only be decrypted by the server, which holds the matching RSA private key.
4. POST the encrypted AES key to /session_key. The server decrypts it, stores it under a fresh session_token, and returns the token.

Storing the keys: Because Python modules are singletons, we save AES_KEY, SESSION_TOKEN and SERVER_PUBLIC_KEY as attributes of the crypto_utils module. Any window (Login, Send File, Admin) can now do import Client.crypto_utils and access these values from RAM, without circular imports or globals.

After the handshake, the standard PyQt initialization runs: QApplication(sys.argv), create the Login window, .show() it, and start the event loop with app.exec_().

WorkerThread Class

class WorkerThread(threading.Thread): def __init__(self, task_queue, signals): super().__init__(daemon=True) self.task_queue = task_queue self.signals = signals def run(self): while True: try: task_name, data = self.task_queue.get() if task_name == "login": self.handle_login(data) except Exception as e: print(f"Error in worker: {e}")
Massive Detailed Explanation:
This class defines the "Worker" who lives in the background. If we didn't have this class, clicking the "Login" button would freeze the entire app until the server responded. With this class, the app stays smooth and the user can even drag the window around while waiting for the login to process.

1. __init__: When we create this worker, we hand it two things: the task_queue (the conveyor belt of jobs) and the signals (the walkie-talkies to report back).
2. daemon=True: This is a critical setting. It means this thread is a "Daemon" (a ghost). If the user suddenly closes the main app window, Windows will instantly kill this background thread too. If it was False, the app window would close, but the ghost thread would keep running invisibly in the background, consuming CPU and causing bugs.
3. run(self): This is the heart of the worker. It contains a while True: loop, meaning it runs forever. It sits at the conveyor belt calling task_queue.get(). This command pauses the thread completely until a job arrives. Once a job arrives (for example, the user clicks login), it unpacks the job name and the data. If the job is named "login", it routes the data to the handle_login function to do the actual checking.

WorkerThread - handle_login (with AES Encryption)

def handle_login(self, data): username, password = data try: # Encrypt the credentials with AES before sending them over the network encrypted = crypto_utils.aes_encrypt({"username": username, "password": password}) r = requests.post(f"{SERVER_URL}/login", json={"session_token": crypto_utils.SESSION_TOKEN, "encrypted_data": encrypted}, timeout=30) response = r.json() if response.get("status") == "Login successful": self.signals.success.emit(username, response.get("is_admin", False)) else: self.signals.error.emit(response.get("status", "Login failed")) except Exception as e: self.signals.error.emit(f"Error: {str(e)}")
Massive Detailed Explanation:
This function performs an encrypted login over the network. Both the username and the password are wrapped in AES-GCM before they ever leave the laptop.

1. username, password = data: Unpacks the tuple that the UI passed in.
2. crypto_utils.aes_encrypt(...): Builds a Python dictionary {"username": ..., "password": ...}, JSON-encodes it, and encrypts it with AES-GCM using the session key established during the handshake. The output is a base64 string like "xK9aP2mLvB..." — completely opaque to anyone listening on the wire.
3. requests.post(f"{SERVER_URL}/login", ...): Sends the encrypted payload to the server. The body of the request contains TWO fields:
- session_token — tells the server which AES key to use for decryption (the server stored it in a dictionary during the handshake).
- encrypted_data — the encrypted credentials.
4. timeout=30: We use a generous timeout because bcrypt password verification is intentionally slow (~250ms) and our cleanup hook also runs against Supabase on every request.
5. Response handling: If the server responds with "Login successful", we shout the success signal with the username and admin flag. Otherwise we shout the error signal so the UI can pop up a red warning. The server's response itself is unencrypted because it contains no sensitive data — only "yes/no" + admin flag.
6. except Exception: If the network is down or the server crashed, we catch the error and emit it to the UI rather than letting the whole client crash.

What an eavesdropper sees on the wire:
{"session_token": "87e25017...", "encrypted_data": "xK9aP2mLvB8fH3rT..."}
No matter how many login attempts they capture, they cannot extract a single password without breaking AES-256.

WorkerSignals Class

class WorkerSignals(QObject): success = pyqtSignal(str, bool) error = pyqtSignal(str)
Massive Detailed Explanation:
This defines the exact "walkie-talkie" channels (Signals) that the background thread is allowed to use to talk to the main thread.

- success = pyqtSignal(str, bool): This creates a signal channel called "success". We strictly define that whoever shouts on this channel MUST send exactly two pieces of data: a String (which will be the username) and a Boolean (True/False, which will be the is_admin status).
- error = pyqtSignal(str): This creates a channel called "error". It requires exactly one String (the error message to display).
By inheriting from QObject, we give this class the ability to hook directly into PyQt5's internal event loop, ensuring messages are passed safely between threads.

3. signup.py (with AES Encryption)

import threading import queue import time import requests from PyQt5 import QtWidgets from PyQt5.QtWidgets import QMainWindow, QMessageBox, QLabel from PyQt5.uic import loadUi from PyQt5.QtCore import Qt, QObject, pyqtSignal from Client.client_config import SERVER_URL from Client import Client_crypto as crypto_utils class WorkerThread(threading.Thread): def __init__(self, task_queue, signals): super().__init__(daemon=True) self.task_queue = task_queue self.signals = signals def run(self): while True: try: task_name, data = self.task_queue.get() if task_name == "signup": self.handle_signup(data) except Exception as e: print(f"Error in worker: {e}") def handle_signup(self, data): username, password = data try: # Encrypt the credentials with AES before sending them over the network encrypted = crypto_utils.aes_encrypt({"username": username, "password": password}) r = requests.post(f"{SERVER_URL}/signup", json={"session_token": crypto_utils.SESSION_TOKEN, "encrypted_data": encrypted}, timeout=30) response = r.json() status = response.get("status", "") if status == "Sign Up successful": self.signals.success.emit(status) else: self.signals.error.emit(status or "Sign Up failed") except Exception as e: self.signals.error.emit(f"Error: {str(e)}") class WorkerSignals(QObject): success = pyqtSignal(str) error = pyqtSignal(str)
Massive Detailed Explanation:
Signup follows the exact same architecture as Login — it uses a background thread to send the user's credentials to the server without freezing the UI, and it now also encrypts the new username and password with AES-GCM before transmission.

Old architecture (removed): Previously, this file directly imported from Server import Database_connection and called handle_signup(username, password) on the same Python process. That meant the client needed a working MySQL connection of its own — which would not work when the client runs on a different physical computer (the school laptop). The whole point of having a server is that only the server talks to the database.

New architecture:
1. from Client import Client_crypto as crypto_utils: Imports the encryption module under an alias so it can be referenced concisely throughout the code.
2. crypto_utils.aes_encrypt({"username": ..., "password": ...}): Wraps the credentials with AES-GCM using the session key established during the handshake (the same session key that protects login).
3. requests.post(f"{SERVER_URL}/signup", json={"session_token": ..., "encrypted_data": ...}): Sends only the session token (an opaque identifier) and the encrypted blob. Anyone sniffing the network sees nothing meaningful.
4. Why does the server still need bcrypt then? Because AES protects the password in transit, but once it arrives, the server decrypts it back to plaintext for verification. Storing that plaintext in the database would be terrible if the DB ever gets stolen — so the server immediately applies bcrypt to produce a one-way hash that lives in MySQL. Encryption (AES) protects the journey; hashing (bcrypt) protects the storage. Both are required.
5. The WorkerThread still uses a Queue + Signals pattern so the UI stays responsive throughout the network call.

Massive Detailed Explanation:
This is the heart of the project's at-rest encryption. Every uploaded file gets its own private AES key, and that key is itself encrypted with the server's RSA public key. The encrypted file ends up in Supabase, the encrypted AES key ends up in MySQL — neither component alone is enough to read the file.

1. select_file: Opens the standard Windows file picker via QFileDialog.getOpenFileName.
2. Recipient Parsing: Splits the comma-separated "To:" box into a list of recipient usernames.
3. Read the file: Opens in "rb" (Read Binary) mode and slurps the raw bytes into raw.
4. Step 1 of Hybrid Encryption — Generate per-file AES key: crypto_utils.generate_aes_key() returns 32 fresh random bytes. This key is used only for this one file. A new key will be generated for the next upload.
5. Step 2 — Encrypt the file with AES-GCM: aes_encrypt_bytes(file_aes_key, raw) produces ciphertext that looks like random noise. Even with the encrypted file in hand, no one can read it without the AES key.
6. Step 3 — Encrypt the AES key with RSA: rsa_encrypt(SERVER_PUBLIC_KEY, file_aes_key) wraps the 32-byte AES key in an RSA-OAEP envelope. Only someone with the server's private RSA key can open this envelope.
7. Build the payload: We base64-encode the encrypted bytes (so they fit in JSON), and we add the new encrypted_aes_key field. filedata now contains ciphertext, not the raw file.
8. Dispatch the background thread: SendFileThread performs the HTTP POST to /upload_file without freezing the UI.

What an attacker would need to read this file: They would need to (a) breach Supabase to get the encrypted bytes, (b) breach our MySQL to get the encrypted AES key, AND (c) breach our running server's RAM to extract the RSA private key. All three. Simultaneously. This is what cryptographers call "defense in depth".

This function solves a massive architectural hurdle. Network printers and PyMuPDF engines are incredibly rigid: they only understand the PDF standard language. They cannot read Microsoft Word (.docx) files. If a user sends a Word document to be printed, we must silently convert it to a PDF behind the scenes before throwing it to the print engine. But converting files, especially those containing Hebrew (Right-To-Left text), is notoriously difficult.

Plan A: The Microsoft Word COM Hook
The absolute best way to convert a Word document is to have actual Microsoft Word do it. We use the docx2pdf library. Under the hood, this library uses Windows COM (Component Object Model) architecture to spawn an invisible instance of the Microsoft Word application in the background, open our file, and trigger Word's native "Export to PDF" function. Because real Word is doing the work, Hebrew text, fonts, and layouts are preserved flawlessly.
- pythoncom.CoInitialize(): This is mandatory. COM objects are tied to specific CPU threads. Because we are running this inside a background worker thread (not the main GUI thread), we must explicitly tell Windows to initialize COM communication for this specific thread, otherwise it crashes immediately.
- We write the bytes to a temporary file, run the conversion, read the new PDF bytes, and aggressively delete (unlink) the temporary files from the hard drive so we don't leak memory.

Plan B: The ReportLab Fallback and the Hebrew Reversal Algorithm
What if the operator's computer doesn't have Microsoft Office installed? Plan A will fail and throw an Exception. We catch that Exception and move to Plan B. We use python-docx to rip the raw text paragraphs out of the Word file, and we use reportlab to manually paint those paragraphs onto a blank PDF canvas.
- The problem: ReportLab does not understand Bidirectional (Bidi) languages. If you give it the word "שלום", it will print it backwards on the PDF as "םולש" from left to right.
- The Solution (fix_hebrew_manual): We wrote a custom algorithm. It uses Regular Expressions (Regex [\u0590-\u05FF]+) to scan the text and identify words containing Hebrew Unicode characters. If it finds Hebrew, it splits the sentence into an array of words. For every Hebrew word, it physically flips the characters backward (w[::-1]). Finally, it flips the order of the words in the entire array, and stitches them back together into a string. We then hand this mangled, backwards string to ReportLab. When ReportLab blindly prints it left-to-right, it "accidentally" comes out perfectly readable Right-To-Left!

Direct Socket Printing (Bypassing Windows)

This is a revolutionary feature. Traditional printing relies on the Windows Print Spooler and proprietary manufacturer drivers. If a driver corrupts, or if you are trying to print to a generic network printer without having the driver installed, printing fails.

This code bypasses Windows entirely. It establishes a raw, direct TCP socket connection (an uninterrupted data pipe) straight into the printer's motherboard via Port 9100.
Because modern commercial network printers contain built-in interpreters that natively understand raw PDF architecture, we do not need to convert the document into a proprietary printer language (like PCL or PostScript).
We take the raw pdf_bytes binary array, and we chop it into manageable chunks of exactly 8,192 bytes (8 Kilobytes). We use a while loop to literally shove these raw bytes directly down the socket pipe using sock.sendall(chunk) until the entire file is transmitted. The printer receives this pure binary stream, recognizes it as a PDF, processes it internally, and mechanically prints it out. This makes the application capable of printing to almost any modern network printer completely driver-free. We emit progress signals along the way so the UI can show a percentage counter.

9. crypto_utils.py — Hybrid Encryption (RSA + AES)

This module powers all encryption on the Client side. It handles two responsibilities: (1) performing the initial RSA/AES handshake with the server when the client boots, and (2) encrypting both messages (login, signup) and entire files before they are uploaded to the cloud, so that nothing sensitive ever leaves the user's computer in plaintext.

Why Hybrid Encryption?

The problem: RSA is wonderfully secure but agonizingly slow. You cannot use it to encrypt a 50MB PDF — it would take many seconds and is mathematically limited to tiny payloads (around 200 bytes per operation). AES, on the other hand, is blazing fast (gigabytes per second) but it is symmetric: both sides must already share the same secret key. So how do two strangers on the internet agree on a secret key without anyone listening in?

The hybrid answer: Use RSA only once to ship a random AES key to the server safely, then use that AES key for all the bulk traffic afterwards. RSA solves the key exchange problem and AES solves the speed problem. This same pattern powers HTTPS, SSH and almost every secure protocol in production today.

Imports & Global State

Massive Detailed Explanation:

1. cryptography: The standard professional Python crypto library, maintained by the Python Cryptographic Authority. Same library used by Mozilla, AWS, OpenStack and countless other production systems.

2. AESGCM: AES in Galois/Counter Mode. GCM is an "authenticated" cipher — it not only encrypts but also creates a tamper-evidence tag. If anyone flips even a single bit on the wire, decryption fails loudly instead of silently producing garbage. This is what protects us against active attackers (not just eavesdroppers).

3. The three module-level variables: These are filled in by main.py right after the handshake. Because Python modules are singletons, any file inside the Client package can simply import Client.crypto_utils and read crypto_utils.AES_KEY — the values stay alive for the entire program lifetime, all in RAM, never on disk.

import_public_key()

Massive Detailed Explanation:

This is human-readable text but the library needs a real Python object to perform encryption with it. load_pem_public_key parses the PEM and returns an RSAPublicKey object that we can call .encrypt(...) on.

generate_aes_key()

def generate_aes_key(): return os.urandom(32)
Massive Detailed Explanation:

AES-256 needs exactly 32 random bytes (256 bits) as its key. os.urandom is the operating system's cryptographically secure random number generator — on Windows it pulls from BCryptGenRandom, on Linux from /dev/urandom.

Critical: We never use Python's random module here. That module is predictable and would let an attacker guess our keys. os.urandom is the only acceptable source.

We call this function twice in our system: once at startup to make the session AES key, and again for every single file upload to make a unique per-file AES key.

rsa_encrypt()

def rsa_encrypt(public_key, data_bytes): encrypted = public_key.encrypt( data_bytes, padding.OAEP( mgf=padding.MGF1(algorithm=hashes.SHA256()), algorithm=hashes.SHA256(), label=None ) ) return base64.b64encode(encrypted).decode()
Massive Detailed Explanation:

This is how the Client safely sends a 32-byte AES key to the server over an insecure network.

1. padding.OAEP: "Optimal Asymmetric Encryption Padding". Never use raw RSA — it has multiple known attacks. OAEP injects randomness before encryption so that encrypting the same input twice produces different ciphertexts.

2. SHA-256 hash function: OAEP needs a hash function internally. SHA-256 must match exactly on both sides (server decrypts with the same SHA-256 OAEP padding).

3. Base64 output: The encrypted bytes are binary garbage. We base64-encode them so they survive a trip through JSON (which is strictly text).

aes_encrypt() — Encrypts a Dictionary (used for messages)

def aes_encrypt(payload_dict): plaintext = json.dumps(payload_dict).encode() nonce = os.urandom(12) aesgcm = AESGCM(AES_KEY) ciphertext = aesgcm.encrypt(nonce, plaintext, None) return base64.b64encode(nonce + ciphertext).decode()
Massive Detailed Explanation:

Called every time we need to send sensitive structured data to the server (login credentials, signup, etc.).

1. json.dumps(...).encode(): Convert the Python dict into a JSON string, then to bytes. AES operates on bytes, not Python objects.

2. nonce = os.urandom(12): A 96-bit "number used once". This must be unique for every single encryption with the same key, otherwise GCM completely breaks. By using fresh random bytes per call, the probability of collision is astronomically small.

3. nonce + ciphertext: We glue the nonce to the front of the ciphertext. The server splits them back apart by taking the first 12 bytes. This way the server doesn't need a separate field for the nonce — it travels with the ciphertext.

4. Result: What an eavesdropper sees on the wire is one big chunk of base64 like "xK9aP2mLvB8fH3rT...". Even if they capture millions of login attempts, they cannot extract a single character of any password without breaking AES-256.

aes_encrypt_bytes() — Encrypts a Raw File

Massive Detailed Explanation:

Identical idea to aes_encrypt() but works on raw bytes instead of a JSON-able dict, and returns raw bytes instead of base64. This is the function we use to encrypt the actual file content before uploading.

Important difference: The AES key here is not the session key. It is a brand-new per-file key generated fresh for each upload via generate_aes_key(). This is the heart of the file-level hybrid encryption.

Massive Detailed Explanation:

The inverse of aes_encrypt_bytes(). Used in sendfile_window.py after a file is downloaded.

1. Split off the 12-byte nonce from the beginning.
2. Feed nonce + ciphertext + AES key to AESGCM.
3. aesgcm.decrypt() verifies the authentication tag and recovers the plaintext bytes in one atomic step. If anyone tampered with the encrypted file (e.g. a malicious Supabase admin), this call throws InvalidTag and we refuse to save the file.

The Handshake — Inside main.py

Massive Detailed Explanation:

This four-step dance is the entire "TLS-like" handshake compressed into Python. It runs once as the first thing the app does, before any window is shown.

1. Pull the public key with a normal unencrypted HTTP GET. Public keys are by definition public, so transmitting in the clear is fine.

2. Generate the AES session key locally. Since we just made it, only we know it.

3. Encrypt the AES key with RSA. The result is a base64 blob that only the server can decrypt (because only the server has the matching RSA private key).

4. Trade the encrypted key for a session token. The server decrypts the AES key, files it under a fresh random token, and returns the token to us. From this moment, we and the server share a secret AES key that no one else in the world knows.

What can a man-in-the-middle do? Even if a hacker captures every packet of this handshake, they only see (1) a public key (which is public anyway) and (2) an AES key encrypted with RSA-2048 (which would take longer than the age of the universe to crack). Game over for the attacker.

File-Level Hybrid Encryption — Uploading a File

Massive Detailed Explanation:

This is the heart of the project's at-rest protection. Each uploaded file gets its own private AES key that nobody else will ever see in plaintext, except the server's RSA private key holder (which exists only in the running server's RAM).

Step-by-step in plain words:
1. The Client reads the entire file from disk into raw (raw bytes).
2. The Client invents a brand-new random AES key just for this one file.
3. The file bytes are encrypted with that key → now they look like random noise.
4. The AES key itself is encrypted with the server's RSA public key → only the server can ever decrypt this.
5. Both encrypted pieces fly up to the server in the same request.

Why is this layered? If someone steals only the Supabase storage, they get a useless encrypted blob — they don't have the AES key. If someone steals only the MySQL database, they get the encrypted AES key — but they don't have the server's RSA private key to decrypt it. They would need to breach all three components simultaneously to read a single file. This is "defense in depth".

End-to-End Cryptographic Summary

Putting it all together, here is the entire cryptographic life cycle of CloudDrop:

1. Startup: Client gets server's RSA public key → Client invents AES session key → encrypts it with RSA → sends to server → server stores it. Both now share a secret session AES key.

2. Login & Signup: Credentials are wrapped with AES-GCM using the session key. The server peels them open with its copy of the same key. Anyone watching the network sees only base64 noise.

3. File Upload: Brand-new per-file AES key invented. File encrypted with that key. The key itself encrypted with the server's RSA public key. Both encrypted pieces stored separately (file in Supabase, key in MySQL). No single component can decrypt anything alone.

4. File Download: Authorized recipient asks for the file. The server uses its RSA private key to recover the file's AES key, then sends both the encrypted file and the AES key back to the recipient over the session-protected channel. The recipient decrypts locally and saves to disk.

🌐 תקשורת ורשתות

שאלה: באיזה פרוטוקול תקשורת אתה עובד ואיך החלטת על זה?
תשובה: אני עובד ב-TCP, ספציפית דרך HTTP (Flask + ספריית requests). מעל זה יש שכבת הצפנה היברידית. בנוסף יש לי TCP socket גולמי להדפסה ברשת על פורט 9100 (פרוטוקול JetDirect). בחרתי ב-TCP כי הוא מבטיח שכל הנתונים יגיעו בשלמותם ובסדר הנכון — קריטי לקבצים, כי אם חצי מה-PDF נאבד הוא לא יפתח.
שאלה: מה ההבדל בין TCP ל-UDP?
תשובה:
• TCP — Transmission Control Protocol. מבטיח שהנתונים יגיעו (Reliable), בסדר הנכון (Ordered), ומבצע לחיצת יד משולשת לפני התקשורת. איטי יותר אבל בטוח.
• UDP — User Datagram Protocol. לא מבטיח הגעה ולא מבטיח סדר. מהיר יותר אבל פחות אמין.
בפרויקט שלי בחרתי ב-TCP כי שליחת קבצים דורשת שלמות מלאה.
שאלה: מהי לחיצת יד משולשת (Three-Way Handshake)?
תשובה: תהליך פתיחת חיבור TCP בשלושה שלבים:
1. SYN — הלקוח שולח בקשה לחיבור.
2. SYN-ACK — השרת מאשר וגם מבקש חיבור חזרה.
3. ACK — הלקוח מאשר. עכשיו החיבור פתוח.
זה קורה אוטומטית כל פעם שאני קורא ל-requests.post(...) או יוצר socket.connect().
שאלה: מהי כתובת IP ולמה משמש PORT?
תשובה:
• IP — כתובת ייחודית למחשב ברשת (לדוגמה 192.168.0.104). היא מזהה איזה מחשב.
• PORT — מספר בין 0-65535 שמזהה איזו אפליקציה במחשב. בפרויקט שלי השרת מאזין על פורט 5000 (Flask) ולמדפסת אני שולח לפורט 9100 (JetDirect).
שאלה: תן דוגמאות לפרוטוקולים בשכבת האפליקציה.
תשובה:
• HTTP — להעברת דפי אינטרנט (משתמשים בפרויקט)
• HTTPS — HTTP מוצפן עם TLS
• SMTP — שליחת מייל
• DNS — תרגום שם דומיין לכתובת IP
• FTP — העברת קבצים
• SSH — חיבור מאובטח לשרת
שאלה: למה משמשת הפקודה PING?
תשובה: לבדיקת זמינות של מחשב ברשת. שולחת חבילות ICMP ומחכה לתגובה. אם חוזרת תגובה — המחשב מחובר וזמין. גם מודדת זמן תגובה.
שאלה: מה זה ARP?
תשובה: Address Resolution Protocol — פרוטוקול שמתרגם כתובת IP לכתובת MAC פיזית בכרטיס הרשת. הכרחי כדי לשלוח חבילות ברשת מקומית.

🔐 קריפטוגרפיה (הצפנה)

שאלה: מהם סוגי ההצפנה הקיימים?
תשובה:
• הצפנה סימטרית — אותו מפתח להצפנה ולפענוח. מהירה אך דורשת חילוף מפתח בטוח. דוגמה: AES (אני משתמש ב-AES-256-GCM).
• הצפנה אסימטרית — שני מפתחות (ציבורי ופרטי). איטית אך פותרת את בעיית חילוף המפתחות. דוגמה: RSA (אני משתמש ב-RSA-2048).
• הצפנה היברידית — שילוב של שניהם. משתמשים ב-RSA כדי להחליף מפתח AES, ואז כל ההודעות מוצפנות עם AES. זה מה שיש לי בפרויקט.
שאלה: מה זה מפתח ציבורי ומה זה מפתח פרטי?
תשובה: בהצפנה אסימטרית יש זוג מפתחות:
• מפתח ציבורי — חופשי להפצה לכולם. מצפינים איתו.
• מפתח פרטי — סודי, רק לבעלים. מפענחים איתו.
בפרויקט שלי, השרת מייצר זוג RSA בעלייה. את המפתח הציבורי הוא שולח לכל לקוח (דרך /public_key). את המפתח הפרטי הוא שומר ב-RAM שלו בלבד. לקוח מצפין מפתח AES עם הציבורי — ורק השרת יכול לפענח אותו עם הפרטי.
שאלה: מה זה Hash ומה ההבדל בינו לבין הצפנה?
תשובה: Hash היא פונקציה חד-כיוונית — לוקחים קלט (כמו סיסמה) ומקבלים פלט קבוע באורך (כמו 64 תווים). אי אפשר להחזיר חזרה. הצפנה לעומת זאת היא דו-כיוונית — אפשר להצפין ולפענח. בפרויקט שלי משתמש ב-bcrypt ל-hash של סיסמאות ב-MySQL, וב-AES להצפנת הסיסמאות בדרך לשרת.
שאלה: מה זו חתימה דיגיטלית?
תשובה: שיטה להוכיח שמסמך מסוים נשלח על ידי מישהו ספציפי ולא שונה בדרך. השולח מצפין hash של ההודעה עם המפתח הפרטי שלו. כל אחד יכול לאמת עם המפתח הציבורי שלו שזה באמת הוא ושההודעה לא שונתה. בפרויקט שלי לא ממומשת חתימה דיגיטלית, אבל AES-GCM נותן אימות הודעות דומה (tampering detection).
שאלה: מה זה MITM (Man-In-The-Middle) ואיך מתמודדים?
תשובה: תקיפה שבה תוקף יושב בין הלקוח לשרת ויכול לקרוא ולשנות את כל מה שעובר. בפרויקט שלי ההצפנה ההיברידית מגנה מזה: גם אם תוקף יושב באמצע — הוא רק יראה bytes מקושקשים (base64) ולא יוכל לקרוא או לשנות בלי להישבר על AES-GCM. בנוסף, AES-GCM מזהה ניסיון לשינוי ומפיל את הפענוח אם משהו לא תקין.

🖥️ מערכות הפעלה

שאלה: מה זה Thread?
תשובה: חוט ביצוע — יחידת ביצוע קטנה בתוך Process. כמה Threads יכולים לרוץ במקביל באותו Process וחולקים את אותו הזיכרון. בפרויקט שלי משתמש ב-QThread של PyQt5 הרבה — לדוגמה SendFileThread, DownloadFileThread, HeartbeatThread — כדי לבצע פעולות רשת ברקע בלי להקפיא את הממשק.
שאלה: מה זה Process?
תשובה: תוכנית פעילה במחשב, עם זיכרון משלה ומשאבי מערכת משלה. בפרויקט שלי השרת רץ ב-Process אחד והלקוח ב-Process אחר (יכולים להיות אפילו על מחשבים שונים). בתוך כל Process יש כמה Threads.
שאלה: מה ההבדל בין Thread ל-Process?
תשובה:
• Process — שטח זיכרון נפרד, מבודד מ-Processes אחרים.
• Thread — חולק את הזיכרון של ה-Process. מהיר יותר ליצירה ולתקשורת בין Threads, אבל דורש סנכרון (locks) כשנוגעים במשאב משותף.
שאלה: מה זה WinAPI?
תשובה: Windows API — אוסף הפונקציות שמערכת ההפעלה Windows חושפת לתוכניות. בפרויקט שלי docx2pdf משתמש ב-WinAPI דרך COM Interface כדי להפעיל את Word מהקוד שלי ולהמיר DOCX ל-PDF. גם QPrinter משתמש ב-WinAPI להדפסה.

📁 קבצים

שאלה: מה זה Magic Number של קובץ?
תשובה: אותם בייטים הראשונים של קובץ שמזהים את סוג הקובץ. לדוגמה:
• PDF מתחיל ב-%PDF (25 50 44 46)
• PNG מתחיל ב-89 50 4E 47
• EXE/DLL מתחיל ב-MZ (4D 5A)
בפרויקט שלי, כשמדפיסים PDF דרך socket למדפסת רשת — המדפסת מזהה את הקובץ לפי ה-magic number שלו.
שאלה: מה זה FAT32?
תשובה: File Allocation Table — מערכת קבצים שמשתמשים בה בעיקר ב-USB ו-SD cards. תומכת בקבצים עד 4GB. דיסקים מודרניים משתמשים ב-NTFS (Windows) או ext4 (Linux).
שאלה: מה זה קובץ PE?
תשובה: Portable Executable — פורמט קבצי ההרצה של Windows (כמו .exe ו-.dll). המבנה כולל header עם metadata, sections של קוד ונתונים, וטבלאות imports/exports.

🛡️ סייבר ואבטחת מידע

שאלה: לאלו מתקפות חשוף הפרויקט שלך ואיך טיפלת?
תשובה:
• MITM — נטרל ע"י הצפנה היברידית (RSA + AES) עם אימות GCM.
• SQL Injection — נטרל ע"י Parameterized Queries בכל הגישות ל-MySQL (%s ולא string concatenation).
• Brute Force — נטרל ע"י נעילת חשבון אחרי 3 כשלונות (lockout של 5 דקות).
• Path Traversal בהעלאת קבצים — נטרל ע"י secure_filename() של Werkzeug.
• גניבת קבצים מהענן — נטרל ע"י הצפנת AES של כל קובץ + הצפנת המפתח ב-RSA.
• גניבת מסד נתונים — סיסמאות מוגנות ב-bcrypt, מפתחות קבצים מוצפנים ב-RSA.
שאלה: מה זה SQL Injection?
תשובה: תקיפה שבה תוקף מזריק קוד SQL זדוני לתוך הקלטים של האפליקציה. לדוגמה אם הסיסמה היא ' OR '1'='1 — וזה משולב ישירות לשאילתה — התוקף יכול לעקוף את ההתחברות. בפרויקט שלי אני משתמש ב-Parameterized Queries (cursor.execute("SELECT ... WHERE username=%s", (username,))) שמפרידים בין הקוד לנתונים.
שאלה: מה זה Buffer Overflow?
תשובה: תקיפה שבה תוקף שולח נתונים גדולים יותר ממה שמערך מסוגל להכיל, וגורם לכתיבה מעבר לגבולות הזיכרון — מה שיכול להחליף את כתובת החזרה של הפונקציה ולגרום להרצת קוד זדוני. בפייתון זה פחות רלוונטי כי הזיכרון מנוהל אוטומטית. בנוסף, יש לי הגבלת גודל קובץ (50MB) שמונעת DoS.
שאלה: איך נשמרת אבטחת הנתונים בפרויקט?
תשובה: ב-3 שכבות:
1. בתעבורה — הצפנה היברידית (RSA + AES-GCM).
2. במסד הנתונים — סיסמאות מוצפנות עם bcrypt + salt, מפתחות קבצים מוצפנים ב-RSA.
3. בענן (Supabase) — קבצים מוצפנים עם AES לפני העלאה, אז גם מי שיחדור לענן לא יוכל לקרוא אותם.

💻 שאלות ספציפיות לפרויקט

שאלה: למה השתמשת בפייתון ולא בשפה אחרת?
תשובה: פייתון נותנת תמיכה מצוינת ב-PyQt5 לממשק, Flask לשרת, ו-cryptography להצפנה. הקוד נקרא בקלות וקיימות ספריות מוכנות לכל מה שצריך — Supabase, bcrypt, PyMuPDF להדפסה. זה איפשר לי להתמקד בלוגיקה ופחות ב-boilerplate.
שאלה: למה Supabase ולא לשמור את הקבצים מקומית?
תשובה: אם הקבצים נשמרים בדיסק של השרת, הם תופסים מקום ואם השרת קורס — מאבדים אותם. בנוסף, מאוד לא מומלץ להחזיק קבצי משתמשים על מחשב אישי. Supabase נותן אחסון בענן בחינם, גישה מכל מקום, ומחיקה אוטומטית. הקבצים בענן מוצפנים — אז גם אם Supabase נפרץ — הקבצים לא קריאים.
שאלה: למה יש לך גם Flask וגם socket בפרויקט?
תשובה: שני סוגי תקשורת שונים:
• Flask (HTTP מעל TCP) — לתקשורת בין הלקוח לשרת. נוח עם JSON ו-endpoints מוגדרים היטב.
• TCP Socket גולמי — להדפסה ישירה למדפסת רשת על פורט 9100. המדפסת לא מבינה HTTP, היא רוצה bytes גולמיים של PDF.
שאלה: למה יש לך הצפנה היברידית במקום רק AES או רק RSA?
תשובה:
• RSA לבד — איטי מדי לקבצים גדולים, ומגביל לכמה מאות בתים בכל הצפנה.
• AES לבד — דורש שהמפתח כבר ידוע בשני הצדדים, אבל איך מעבירים אותו בצורה בטוחה?
• הצפנה היברידית — RSA פותר את חילוף המפתחות (פעם אחת בלבד), ואז AES מהיר עושה את העבודה הקשה.
שאלה: מה קורה אם השרת קורס באמצע handshake?
תשובה: הלקוח מקבל ConnectionError מ-requests. ה-try/except במקום הזה תופס את החריגה ומציג הודעת שגיאה למשתמש. ה-AES key לא נוצר, אז הלקוח לא יכול לבצע login עד שהשרת חוזר וה-handshake מתבצע מחדש.
שאלה: למה לא השתמשת ב-HTTPS במקום הצפנה יישומית?
תשובה: HTTPS דורש תעודות SSL חתומות ע"י Certificate Authority. בסביבה לימודית/בית-ספרית אין לי גישה לכך. ההצפנה ההיברידית שמימשתי משיגה אותה רמת אבטחה (RSA-2048 + AES-256-GCM זה בדיוק מה ש-TLS משתמש בו), בלי תלות בתעודות.
שאלה: איך עובד QThread? למה צריך אותו?
תשובה: Qt מגביל את הגישה ל-UI ל-Thread הראשי בלבד. אם הייתי שולח קובץ של 50MB ב-Thread הראשי — הממשק היה קופא לכמה שניות. במקום זה, אני יוצר QThread שעושה את ההעלאה ברקע, ומעדכן את ה-UI דרך pyqtSignal. ה-Signals הם הדרך הבטוחה היחידה לתקשר בין Thread רקע ל-Thread של ה-UI.

💡 Glossary of Key Concepts & Technologies

This table breaks down the core technologies, libraries, and programming concepts used throughout the CloudDrop Client to make everything work seamlessly.

Technology / Concept Massive Detailed Explanation
Base64 Encoding What it is: A way to translate raw computer binary (1s and 0s) into normal English text.
Why we use it: When sending a physical file (like a PDF or JPG) over the internet using JSON, you cannot send raw binary because it corrupts the text formatting of the JSON. Base64 mathematically converts the file into a massive, safe string of letters and numbers (A-Z, 0-9). The server receives this text, and "decodes" it back into the exact original physical file.
QThread & Multithreading What it is: A PyQt5 tool that allows the computer to do multiple tasks at the exact same time.
Why we use it: A graphical app runs on a "Main Thread". If the main thread stops to wait for a 50MB file to download, the entire screen freezes and turns white (Not Responding). By putting heavy tasks (like internet downloads, or scanning 254 IP addresses) into a QThread, a "ghost worker" does the heavy lifting in the background, keeping the main app screen 100% smooth and clickable.
TCP Sockets What it is: A direct, low-level internet pipe between two computers (or a computer and a printer).
Why we use it: Normally, printing requires complex Windows Drivers. By using a raw TCP Socket on Port 9100, we bypass Windows entirely. We shove the raw PDF data directly down the socket pipe straight into the printer's motherboard. The printer natively understands the PDF and prints it instantly. It's incredibly fast and doesn't require driver installations.
PyMuPDF (fitz) What it is: A high-performance Python library for reading and rendering PDF files.
Why we use it: To show a preview of a document before printing, or to send a document to the Windows Print Spooler, we need a picture of the document. PyMuPDF takes the raw PDF code and "rasterizes" it—meaning it draws a massive, high-resolution (300 DPI) picture of the page. We then paint this picture onto the screen or the printer.
COM Interface (pythoncom) What it is: Component Object Model (COM) is a Microsoft technology that lets Python scripts remote-control Windows applications.
Why we use it: We use it via the docx2pdf library to silently remote-control Microsoft Word in the background. It forces Word to open our .docx file and save it as a .pdf. Because we are using the real MS Word engine, complex languages like Hebrew (Right-To-Left) format absolutely perfectly on the final PDF.
JSON (JavaScript Object Notation) What it is: A universal text format for sending structured data over the internet.
Why we use it: Whenever we talk to our Flask server, we pack our data into a Python Dictionary (like {"sender": "Daniel", "message": "Hello"}). This is translated into JSON format so it can be cleanly transmitted through HTTP requests (using the requests library).
Signals and Slots (pyqtSignal) What it is: The communication system used inside PyQt5.
Why we use it: Background QThreads are not allowed to touch the UI (like changing a label's text), because it will crash the app. Instead, the background thread emits a "Signal" (like a walkie-talkie shout). The Main UI thread connects a "Slot" (a function) to listen for that shout. When it hears the signal, the main thread safely updates the screen.
QCompleter What it is: A PyQt5 widget that provides an auto-complete dropdown menu.
Why we use it: In the Send File window, when you start typing a recipient's name, we don't want you to have to type the whole thing. The QCompleter takes our downloaded list of users and automatically suggests matching names as you type, making the app feel incredibly modern and fast.
